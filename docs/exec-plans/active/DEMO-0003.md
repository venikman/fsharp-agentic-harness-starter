# Execution plan

## Issue

`DEMO-0003`

## Goal

Replace the repo-copy bootstrap with a deterministic git-based workspace provisioning strategy while preserving the hook boundary and keeping all writes inside the configured workspace root.

## Recommendation basis

This plan follows the shared harness recommendations that per-issue workspaces should be isolated, deterministic, and repo-controlled, while repo-specific setup remains outside the orchestrator core.

Key principles:
- workspaces are disposable execution environments, not the system of record
- workspace provisioning belongs at the workspace/bootstrap seam, not inside generic orchestration logic
- local-first trust assumptions remain explicit
- no remote-host or container assumptions are introduced in this pass

## Prerequisites

- `DEMO-0002` should be complete first.
- Reproduce the current `tools/AfterCreate.fsx` behavior and understand what files are intentionally excluded today.

## Bounds

- in scope:
  - replace repo-copy bootstrap in `tools/AfterCreate.fsx`
  - choose and document a git-backed strategy such as `git worktree` or a clone-based fallback
  - define the sync/reset behavior for reused workspaces
  - leave clear evidence when bootstrap fails
- out of scope:
  - remote execution
  - containerization
  - changing generic workspace path invariants in `DeliveryHarness.Core` unless strictly required
  - making the orchestrator responsible for repository sync policy

## Detailed steps

1. **Document the baseline bootstrap behavior**
   - Review the current copy-based hook and the excluded-path set.
   - Identify what behavior must still be preserved after the switch:
     - workspace-local `.harness/` directory
     - no writes to the repo root during bootstrap
     - failure surfaces visible to the operator

2. **Choose the first git-backed strategy**
   - Prefer the simplest strategy that matches the repository’s local-first posture.
   - Evaluate:
     - `git worktree` for fast, deterministic workspace creation
     - shallow/local clone fallback only if worktree is unsuitable
   - Record the choice and the reason in docs and/or hook comments.

3. **Define reused-workspace policy before coding**
   - Decide what happens when a workspace already exists:
     - leave it untouched
     - verify it points at the expected repo state
     - optionally perform a safe sync/reset step if documented
   - Keep the policy explicit and conservative; do not silently destroy work.

4. **Implement the hook-only bootstrap change**
   - Keep repo-specific provisioning in `tools/AfterCreate.fsx`.
   - Ensure all paths are resolved relative to the assigned workspace and repo root inputs.
   - Write a clear stamp/evidence file under workspace `.harness/` so bootstrap outcomes are inspectable.

5. **Cover failure modes and operator evidence**
   - Handle missing git metadata, invalid repo roots, or failed worktree/clone commands explicitly.
   - Ensure failures surface to the hook caller with actionable messages.
   - Confirm the repo root is not mutated when bootstrap fails partway through.

6. **Update docs and issue guidance**
   - Document the chosen bootstrap strategy and workspace reuse policy.
   - Keep the hook-driven boundary explicit in architecture docs if the implementation choice changes the operator contract.

7. **Validate with an issue run**
   - Build the CLI project.
   - Run `DEMO-0003` through the harness.
   - Inspect the created workspace and `.harness/after-create.txt` or replacement evidence.

## Expected deliverables

- new git-backed `tools/AfterCreate.fsx`
- documented workspace reuse/reset policy
- clear workspace-local evidence for bootstrap success/failure
- updated docs if the operator contract changes

## Risks

- risk: worktree/bootstrap logic accidentally mutates the repo root
  - mitigation: keep all provisioning logic scoped to the workspace and fail fast on path mistakes
- risk: reused workspaces become implicitly reset and destroy useful state
  - mitigation: document and implement a conservative reuse policy first
- risk: git assumptions fail in non-standard local setups
  - mitigation: emit explicit failure messages and keep the strategy replaceable
