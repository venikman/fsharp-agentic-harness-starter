# Execution plan

## Issue

`DEMO-0002`

## Goal

Restore a trustworthy baseline by making the starter build cleanly and by adding deterministic automated tests for the current local-first harness contract.

## Recommendation basis

This issue is the first gate because the shared harness-engineering recommendations and the Symphony spec assume a harness that can be validated locally before autonomy expands. For this repository, that means:

- repo-owned docs and `WORKFLOW.md` remain the system of record
- the local file-backed/dry-run path stays usable while the harness matures
- evidence and deterministic checks come before deeper orchestration changes
- no new config surface is introduced outside harness-owned files

Relevant conformance targets from the Symphony spec:
- workflow/config parsing
- workspace safety
- orchestration basics that can be tested without a real tracker or real agent
- implementation-checklist emphasis on a trustworthy local proving path before broader rollout

## Bounds

- in scope:
  - reproduce the current failing build and record the baseline
  - fix the smallest code defects needed to restore a green build
  - add one deterministic automated test project for the harness
  - add fixtures for workflow parsing, issue parsing, workspace safety, and run-record behavior
  - update docs/validation references so future issues can rely on `dotnet test`
- out of scope:
  - real tracker integration
  - long-running orchestrator mode
  - workflow hot reload
  - replacing dry-run as the default path
  - introducing `.pi/` settings or any non-harness config surface

## Detailed steps

1. **Capture the failing baseline**
   - Run the documented restore/build commands.
   - Record the exact build error and warnings.
   - Confirm whether the failure is limited to compile-time issues or whether runtime smoke checks are also currently blocked.

2. **Fix the build with the smallest credible change set**
   - Inspect `src/DeliveryHarness.Core/Agent.fs` around the `FS0072` failure and add the minimum type clarification or refactor needed.
   - Decide whether the `ProcessRunner.fs` reserved-identifier warnings should be cleaned up in this issue.
     - If they are trivial and low-risk, fix them now.
     - If not, record them explicitly so they do not disappear into background noise.
   - Keep behavior changes minimal; this issue is about baseline trust, not new features.

3. **Create a deterministic test project**
   - Add a single harness test project under `tests/`.
   - Reference `DeliveryHarness.Core` directly.
   - Choose the smallest F#-appropriate test dependency footprint that keeps the suite easy to run locally.
   - Ensure the test project runs without network access or external credentials.

4. **Add fixture strategy before broad test coverage**
   - Create small markdown fixtures for:
     - valid workflow front matter and prompt body
     - valid issue files with acceptance/validation/constraints
     - malformed or edge-case front matter when useful
   - Use temporary directories for workspace and run-record tests so the suite does not mutate real repo state.
   - Keep timestamps, paths, and generated identifiers deterministic where possible.

5. **Implement the first conformance slice**
   - Add tests for `Workflow.load` and `Workflow.validate` defaults/basic errors.
   - Add tests for file-backed issue parsing and sorting.
   - Add tests for workspace identifier sanitization and workspace-root containment.
   - Add tests for run-record writing and/or dry-run artifacts at the smallest seam that avoids over-mocking.
   - Prefer pure/module-level tests first; only add heavier integration-style tests when a pure seam is not enough.

6. **Align repo docs with the new baseline**
   - Update `docs/QUALITY.md` only where the new test entry point or evidence expectations need clarification.
   - If needed, update README quick-start guidance so agents and humans know the baseline validation order.
   - Keep the new guidance harness-owned; do not move policy into user-local tooling config.

7. **Run the validation set and capture evidence**
   - `dotnet restore src/DeliveryHarness.Cli/DeliveryHarness.Cli.fsproj`
   - `dotnet build src/DeliveryHarness.Cli/DeliveryHarness.Cli.fsproj`
   - `dotnet test`
   - `dotnet run --project src/DeliveryHarness.Cli/DeliveryHarness.Cli.fsproj -- validate-workflow`
   - If feasible after the build is restored, optionally run `list-issues` as an extra smoke check.

8. **Leave the repo ready for Wave 1**
   - Summarize what the tests now protect.
   - List remaining untested seams explicitly so later issues know where they still need caution.
   - Confirm that no new `.pi/` config dependency was introduced.

## Expected deliverables

- green build on the documented .NET SDK
- a new deterministic test project under `tests/`
- fixture files or helper builders for starter-harness inputs
- updated quality/validation docs where needed
- review handoff notes that explain what baseline risks were removed and what risks remain

## Risks

- risk: fixing the build drifts into feature work
  - mitigation: keep code changes limited to restoring the documented starter behavior
- risk: tests rely on the live repo layout and become brittle
  - mitigation: prefer dedicated fixtures and temp directories
- risk: time-based or path-based behavior makes tests flaky
  - mitigation: control temp paths and assert on stable structure rather than incidental formatting
- risk: too much abstraction is introduced just to make tests pass
  - mitigation: test existing seams first and only extract helpers when there is a clear repeated need
