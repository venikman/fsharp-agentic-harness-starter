# Execution plan

## Issue

`DEMO-0007`

## Goal

Provide operator-visible structured logs and a lightweight runtime status surface without making observability a correctness dependency.

## Recommendation basis

This plan follows the shared recommendation that a harness must leave enough observability to operate and debug multiple runs, while keeping optional status surfaces separate from core correctness.

Key principles:
- startup, validation, dispatch, and failure states must be visible without a debugger
- logs should carry stable issue/run/session context
- optional status surfaces must reflect orchestrator state, not become a second source of truth
- secrets and large raw payloads should not leak into logs

## Prerequisites

- `DEMO-0005` should establish long-running runtime state worth observing.
- Reuse existing run records where they help, but do not confuse static run history with live status.

## Bounds

- in scope:
  - add structured logging for startup, validation, dispatch, retries, hooks, and failures
  - include stable issue/run context fields in logs
  - add a status command or machine-readable snapshot for current runtime state
  - document log/snapshot semantics
- out of scope:
  - a mandatory dashboard
  - large telemetry systems or remote sinks
  - logging secrets or full raw agent payloads by default

## Detailed steps

1. **Define the minimum observability contract**
   - Decide what fields every issue-related log should include.
   - Decide what live status must expose:
     - running work
     - retrying work
     - attempt counts and timing where available
   - Keep the surface intentionally small and operator-oriented.

2. **Add structured logging at the orchestration seams**
   - Emit logs for:
     - startup validation success/failure
     - issue dispatch decisions
     - hook execution start/failure/timeout
     - worker success/failure/timeout
     - retry scheduling and exhaustion
     - reconciliation stop/cleanup events
   - Use stable key/value or similarly machine-readable formatting.

3. **Protect logs from becoming a leak path**
   - Avoid raw secret values.
   - Avoid logging full request bodies or unbounded stdout/stderr payloads.
   - If truncation or summarization is applied, make it documented and deterministic.

4. **Add a lightweight status surface**
   - Choose the smallest useful surface for this stage, such as:
     - a CLI `status` command
     - a machine-readable snapshot file or JSON output
   - Ensure it is derived from orchestrator state only.
   - Keep it optional; the host must still work if no one reads the status output.

5. **Test observability behavior**
   - Verify key log events are emitted for representative happy/failure paths.
   - Verify the status/snapshot surface reflects current running and retry state.
   - Verify failures in optional status generation do not crash the orchestrator.

6. **Document operator expectations**
   - Explain where logs go, what the fields mean, and how to read current runtime status.
   - Keep any future dashboard work explicitly optional.

## Expected deliverables

- structured logging around core orchestration events
- stable context fields for issue/run correlation
- a lightweight status or snapshot surface
- tests or deterministic checks for observability outputs
- docs describing how operators inspect the harness at runtime

## Risks

- risk: observability code becomes coupled to orchestration correctness
  - mitigation: keep logs/status as projections of state, never the state itself
- risk: logs become noisy or leak sensitive data
  - mitigation: define explicit field sets and redaction/truncation rules
- risk: a snapshot surface grows into an accidental control plane
  - mitigation: keep the first version read-only and minimal

## Validation

- `dotnet test`

## Handoff expectations

A review-ready result should include:
- example log lines or snapshot output
- the documented field set used for issue/run context
- notes on any redaction/truncation choices
- confirmation that the host still functions without an optional UI or dashboard
