# Execution plan

## Issue

`DEMO-0011`

## Goal

Define the harness meaning of attempts and turns so `agent.max_turns` has real, documented semantics without locking the harness to a single coding-agent vendor or protocol.

## Recommendation basis

This plan follows the shared recommendation that the harness should own its orchestration contract explicitly instead of exposing configuration fields whose runtime meaning is ambiguous.

Key principles:
- configuration should reflect actual behavior, not aspirational behavior
- the worker contract must remain generic enough for multiple agent runtimes
- attempt/turn semantics belong at the harness/runtime boundary, not in vendor-specific assumptions
- the dry-run proving path must remain supported

## Prerequisites

- `DEMO-0001` should establish the real worker-command path.
- `DEMO-0005` should establish long-running orchestration and retries.
- Reproduce the current meaninglessness of `agent.max_turns` after those issues land.

## Bounds

- in scope:
  - define what one attempt means in the generic worker model
  - define what one turn means, or explicitly reject the concept where unsupported
  - make `agent.max_turns` either enforced or validation-failed when unsupported
  - record enough metadata to distinguish attempts from turns
  - update docs and tests accordingly
- out of scope:
  - vendor-specific app-server integration unless needed as a documented optional path
  - locking the harness to pi, Codex, or another single runtime
  - speculative multi-agent workflows

## Detailed steps

1. **Write the runtime vocabulary first**
   - Define the difference between:
     - issue run
     - attempt
     - turn
     - retry/continuation
   - Keep the vocabulary small and operator-readable.

2. **Choose the generic CLI semantics**
   - Decide whether a generic external command invocation represents:
     - exactly one turn, with no internal turn control by the harness
     - or one attempt containing bounded continuations orchestrated by the harness
   - Prefer the smallest model that the harness can enforce and test reliably.

3. **Make `agent.max_turns` honest**
   - If the generic worker path can support bounded continuation behavior, implement it and test it.
   - If it cannot, make workflow validation reject unsupported configurations clearly.
   - Avoid keeping a config field that suggests behavior the runtime does not provide.

4. **Record attempt/turn metadata**
   - Update run records or runtime state so operators can see whether a failure or success happened on:
     - first attempt versus retry
     - first turn versus later turn, if that concept exists
   - Keep the metadata simple enough for logs, status surfaces, and future prompt templates.

5. **Test the contract**
   - Add tests for validation behavior around `agent.max_turns`.
   - Add tests for attempt/turn metadata emission.
   - If continuation behavior exists, test stop conditions and boundaries explicitly.

6. **Document the chosen model**
   - Explain how the harness interprets `agent.max_turns`.
   - Explain what generic CLI mode can and cannot do.
   - Keep the contract vendor-neutral.

## Expected deliverables

- documented attempt/turn vocabulary
- honest runtime behavior for `agent.max_turns`
- run/status metadata that distinguishes attempts from turns where applicable
- tests proving the chosen model
- updated docs for operators and later prompt work

## Risks

- risk: the harness keeps a misleading config field for compatibility reasons
  - mitigation: prefer explicit validation failure over ambiguous semantics
- risk: the model becomes vendor-specific too early
  - mitigation: define the contract in generic worker terms first
- risk: continuation behavior complicates retries and status surfaces
  - mitigation: keep the first model minimal and coordinated with orchestration/state docs

A review-ready result should include:
