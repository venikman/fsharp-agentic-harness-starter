## Why this issue exists

The current repository now contains the harness-only capability required for the local file-backed path, but it still needs one deliberate trusted-local release-proof pass. This issue acts as the top-level programme control surface that answers:

- what “done” means for the harness as a whole,
- which lower-level issues are mandatory before autonomous execution is credible,
- what runtime behavior must exist before a single-command start is trustworthy,
- and what evidence proves the final end-to-end claim.

## Bounds

- in scope:
  - define the final project-level completion criteria
  - define the prerequisite issue set for one-start autonomous execution
  - define the required issue-state transition policy
  - define the final end-to-end smoke validation needed to claim success
- out of scope:
  - implementing all prerequisite issues inside this issue
  - bypassing lower-level issue dependencies
  - introducing non-harness config surfaces

## Program-level done definition

The harness is ready for “start once, finish the expected work” only when all of the following are true:

1. **Baseline trust exists**
   - the harness builds and tests cleanly
   - workflow validation is green

2. **Worker/runtime seam exists**
   - a real external worker command path exists
   - dry-run still exists as a proving path when needed
   - request-file and transcript behavior are deterministic

3. **Workspace seam exists**
   - workspace provisioning is git-backed or otherwise documented as the real intended bootstrap path
   - reused-workspace behavior is explicit

4. **Tracker seam exists**
   - orchestration depends on a tracker abstraction rather than file-tracker internals
   - file-backed issue intake still works as the local proving path

5. **Safety baseline exists**
   - workspace containment, hook/process failure rules, and secret-handling rules are enforced and documented

6. **Long-running orchestration exists**
   - a long-running host command exists
   - bounded concurrency and retry policy are real runtime behavior
   - reconciliation against tracker state is real runtime behavior
   - operator-visible logs/status exist
   - workflow reload behavior is documented and supported where intended

7. **Runtime contract is honest**
   - attempt/turn semantics are explicit
   - prompt templating is strict and aligned to actual runtime state

## Required issue transition policy

Before this issue is considered complete, the repository must have one explicit policy for:

- when an issue becomes `In Progress`
- who or what is allowed to mark an issue review-ready
- how blockers are represented
- what terminal states exist for this repo’s real workflow
- how retries relate to issue state
- whether issue-state changes are runtime-automated, human-operated, or hybrid

This policy may live in `WORKFLOW.md`, repo docs, or both, but it must be harness-owned and operator-readable.

## Final end-to-end validation

Once prerequisites are done, run a full trusted-local smoke validation and fill in the release checklist:

1. Build and test the harness.
2. Validate `WORKFLOW.md`.
3. Start the long-running host command once.
4. Let it process a controlled set of eligible issues.
5. Confirm that the resulting outcome is one of:
   - the expected completed project-level work with evidence, or
   - a bounded blocker report with evidence and next-step clarity.

Expected evidence set:
- build output
- test output
- workflow validation output
- host-mode runtime logs/status
- workspace `.harness/*` artifacts for the exercised issues
- final run records under `.harness/runs/`
- issue-state outcome summary
- completed `docs/exec-plans/active/DEMO-9999-RELEASE-CHECKLIST.md`
