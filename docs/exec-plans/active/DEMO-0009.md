# Execution plan

## Issue

`DEMO-0009`

## Goal

Add a Linear-compatible read adapter for the tracker seam while preserving the file-backed tracker as the default proving path and keeping credentials in environment-backed harness config.

## Recommendation basis

This plan follows the shared harness guidance that external integrations should come after the local proving path is stable, and that tracker adapters should plug into the orchestrator through a read-oriented boundary rather than embedding vendor details in core orchestration logic.

Key principles:
- local file-backed proof remains available
- external credentials come from harness-managed environment variables
- tracker reads are sufficient for this pass
- trust assumptions and environment requirements must be documented explicitly

## Prerequisites

- `DEMO-0004` should establish the tracker seam.
- `DEMO-0010` should harden secret/process handling before external credentials and network calls are introduced.
- Prefer finishing Wave 2 first so the harness has stronger observability when real integration arrives.

## Bounds

- in scope:
  - add a Linear-compatible read adapter behind the tracker seam
  - support candidate issue fetch, issue-state refresh, and terminal-issue listing
  - read credentials from environment-backed workflow config
  - preserve the markdown file tracker for local tests and proofing
  - document environment and trust assumptions
- out of scope:
  - tracker writes such as comments, transitions, or PR links
  - replacing the file-backed tracker
  - hidden credential files checked into the repo

## Detailed steps

1. **Freeze the adapter contract before implementing network code**
   - Reconfirm the exact tracker-seam methods expected by the orchestrator.
   - Ensure the Linear-compatible adapter satisfies only those methods.
   - Keep the normalized issue model harness-owned.

2. **Define configuration and secret handling**
   - Add workflow config fields only where needed for tracker kind/project selection.
   - Resolve API tokens through harness-managed environment variables.
   - Validate presence of required secrets without printing them.

3. **Implement the read adapter**
   - Support:
     - fetch candidate issues in configured active states
     - refresh issue state for known ids
     - fetch terminal issues for cleanup/reconciliation scenarios
   - Normalize external payloads into the stable tracker issue model.
   - Keep vendor-specific transport and payload handling inside the adapter.

4. **Add failure mapping and deterministic tests**
   - Cover request failures, malformed payloads, and missing configuration.
   - Prefer deterministic unit tests with stubbed payloads for the default suite.
   - Keep any live integration checks optional and explicitly gated.

5. **Preserve the file-backed fallback**
   - Ensure `tracker.kind: file` still works exactly as the proving path.
   - Keep local issue-driven development and tests independent of network access.

6. **Document trust and operating assumptions**
   - Update docs with required environment variables, project/team assumptions, and the fact that this pass is read-only.
   - Clarify that tracker writes, comments, and state transitions remain outside the orchestrator core.

## Expected deliverables

- Linear-compatible tracker adapter behind the existing seam
- env-backed configuration for required credentials
- deterministic unit tests for adapter behavior and failure mapping
- preserved file-backed tracker behavior
- updated security/usage docs for external tracker access

## Risks

- risk: vendor-specific details leak into core orchestration types
  - mitigation: keep normalization inside the adapter
- risk: live integration becomes required for the default test suite
  - mitigation: keep network-free tests as the default and gate live checks explicitly
- risk: secret handling is too permissive or too opaque
  - mitigation: validate presence, redact values, and document assumptions clearly

## Validation

- `dotnet build src/DeliveryHarness.Cli/DeliveryHarness.Cli.fsproj`
- `dotnet test`

## Handoff expectations

A review-ready result should include:
- the new workflow/env configuration required for Linear-compatible reads
- confirmation that the file-backed tracker still works
- test evidence for adapter normalization and failure mapping
- clear notes on what remains intentionally out of scope, especially tracker writes
