# Execution plan

## Issue

`DEMO-0006`

## Goal

Make tracker state the source of truth for live-run reconciliation so terminal and non-active issues are handled correctly during long-running orchestration.

## Recommendation basis

This plan follows the shared harness recommendations and Symphony-style runtime model that active runs must be reconciled continuously against tracker state.

Key principles:
- the orchestrator owns running-state reconciliation
- tracker state, not stale in-memory assumptions, determines whether work should continue
- terminal cleanup is policy-driven and explicit
- tracker reads belong in the harness; tracker writes remain out of scope in this pass

## Prerequisites

- `DEMO-0005` should provide the long-running orchestration state and host loop.
- `DEMO-0004` should provide the tracker seam needed for state refresh and terminal queries.
- Reuse the existing terminal-cleanup policy switch instead of inventing a new one.

## Bounds

- in scope:
  - refresh running issues against current tracker state during each tick
  - stop and clean terminal issues according to policy
  - stop non-active, non-terminal issues without destructive cleanup
  - preserve or refine startup terminal cleanup behavior
  - add tests for reconciliation decisions
- out of scope:
  - tracker writes
  - advanced host failover or distributed worker coordination
  - dashboard/UI work beyond what later observability issues need

## Detailed steps

1. **Make the reconciliation contract explicit**
   - Define the exact states the host cares about:
     - active
     - terminal
     - neither active nor terminal
   - Document what each transition means for:
     - the running worker
     - workspace cleanup
     - retry/claim state

2. **Add tracker refresh for running work**
   - Use the tracker seam to fetch current state for running issue ids.
   - Keep refresh logic best-effort: failed refresh should not crash the host.
   - Ensure the orchestrator updates its in-memory issue snapshot when the state remains active.

3. **Implement terminal-state handling**
   - When a running issue becomes terminal:
     - stop active work
     - clean the matching workspace if policy allows
     - release claims/retry state appropriately
   - Preserve workspace cleanup policy as workflow-controlled behavior.

4. **Implement non-active, non-terminal handling**
   - When a running issue leaves the active set but is not terminal:
     - stop active work
     - do not destroy the workspace
     - leave operator-visible evidence so the stop is explainable

5. **Review startup cleanup behavior**
   - Align startup cleanup with the new reconciliation contract.
   - Keep startup cleanup conservative and policy-driven.
   - Avoid duplicating cleanup logic in multiple inconsistent places.

6. **Test reconciliation decisions deterministically**
   - Add tests for:
     - active issues remaining active
     - terminal transitions causing stop + cleanup
     - paused/review/non-active transitions causing stop without cleanup
     - failed state refresh leaving workers alone until the next tick
   - Prefer fake tracker responses and a controllable worker/run abstraction.

7. **Document the operator contract**
   - Clarify what happens when an issue moves to `Done`, `Closed`, `Cancelled`, or another non-active state.
   - Keep the explanation harness-owned and consistent with workflow-configured state sets.

## Expected deliverables

- reconciliation logic for running issues in the host loop
- correct terminal cleanup behavior tied to workflow policy
- correct non-active stop behavior without destructive cleanup
- deterministic tests for reconciliation cases
- docs clarifying live-run behavior

## Risks

- risk: cleanup behavior becomes destructive for non-terminal states
  - mitigation: keep terminal and non-active paths explicitly separate
- risk: failed tracker refresh kills or mutates live work incorrectly
  - mitigation: treat refresh failure as best-effort and retry next tick
- risk: retry state and reconciliation state interact unexpectedly
  - mitigation: write tests around claim/retry release rules for state transitions
