# Execution plan

## Issue

`DEMO-0005`

## Goal

Add a long-running orchestrator mode that owns scheduling, bounded concurrency, and retry backoff while preserving `poll-once` as the simple proving/debugging path.

## Recommendation basis

This plan follows the shared recommendations and Symphony spec emphasis that the harness should eventually act as a long-running scheduler/runner with a single authoritative runtime state, while still starting from a thin, local-first posture.

Key principles:
- orchestration belongs in the harness, not in the worker runtime
- bounded concurrency and retry state are harness concerns
- restart recovery can remain simple and tracker-driven in early phases
- the service must not pretend to be a hardened control plane before the surrounding safety/observability work exists

## Prerequisites

- `DEMO-0002` should be complete first.
- `DEMO-0004` should establish the tracker seam before long-running orchestration depends on it.
- Reuse `poll-once` behavior as the baseline semantics wherever possible instead of inventing a second dispatch model.

## Bounds

- in scope:
  - add a long-running host/daemon-style CLI mode
  - maintain in-memory orchestrator state for claimed/running/retrying work
  - enforce `orchestrator.max_concurrency`
  - make retry backoff and `orchestrator.max_attempts` real behavior
  - preserve `poll-once` for debugging
- out of scope:
  - distributed orchestration
  - persistent retry queues or databases
  - tracker writes
  - pretending the system is production-hardened before later waves

## Detailed steps

1. **Specify the long-running state model before implementation**
   - Define the minimum runtime state needed:
     - running work
     - claimed work
     - retry attempts/backoff state
     - current effective workflow config values used by the host
   - Keep the state model small and aligned with the current local-first trust posture.

2. **Add a new CLI host mode**
   - Introduce a command such as `serve`, `run-loop`, or similarly explicit host mode.
   - Preserve existing one-shot commands untouched where possible.
   - Make startup validation fail clearly if the workflow is invalid.

3. **Lift `poll-once` semantics into the host loop**
   - Start from the current polling logic rather than rewriting from scratch.
   - Schedule repeated ticks using `orchestrator.poll_interval_seconds`.
   - Keep `poll-once` as a one-shot wrapper around the same core scheduling logic where feasible.

4. **Implement bounded concurrency and claims**
   - Prevent duplicate dispatch of the same issue while it is already running or reserved for retry.
   - Enforce `orchestrator.max_concurrency` using the new runtime state.
   - Be explicit about how the host handles slot exhaustion.

5. **Make retry behavior real**
   - Define how a failed attempt increments retry count.
   - Implement documented backoff rules for the current local-first host.
   - Stop retrying after `orchestrator.max_attempts` is reached and leave operator-visible evidence.
   - Keep the retry model simple enough to reason about without a database.

6. **Test the scheduling contract**
   - Add deterministic tests for:
     - max-concurrency enforcement
     - duplicate-claim prevention
     - retry scheduling/backoff decisions
     - attempt cap enforcement
   - Prefer fake or in-memory tracker/worker seams over slow integration tests.

7. **Document the runtime contract**
   - Update docs so operators know when to use `poll-once` versus the long-running host mode.
   - Keep the trust posture explicit: this is still a local/trusted-environment harness unless later issues harden it.

8. **Run validation and capture host evidence**
   - Build and test the harness.
   - If possible, run a bounded local host smoke check that exercises at least one retry or concurrency decision in a controlled way.

## Expected deliverables

- new long-running CLI host mode
- in-memory orchestrator state for scheduling and retries
- effective enforcement of max concurrency and max attempts
- preserved `poll-once` path
- tests for the host scheduling contract

## Risks

- risk: host mode diverges semantically from `poll-once`
  - mitigation: share the same core dispatch logic where possible
- risk: retry logic becomes overly complex before reconciliation exists
  - mitigation: keep the first host loop simple and aligned with current issue semantics
- risk: the system overstates its safety posture
  - mitigation: document the trusted-local boundary clearly in code and docs
