# Execution plan

## Issue

`DEMO-0005`

## Status

Completed baseline reference retained until plan archival exists.

## Outcome

`DeliveryHarness.Cli` now exposes `serve`, and `DeliveryHarness.Core/Orchestrator.fs` maintains in-memory running/retrying/retired state so host mode can enforce `orchestrator.max_concurrency`, schedule retries with linear backoff, and preserve `poll-once` as the one-shot debugging path.

## What later work should assume

- `serve` is the canonical long-running host command
- retries are real runtime behavior driven by `orchestrator.max_attempts`
- successful or exhausted unchanged active issues are not redispatched until tracker state changes
- `poll-once` remains one-shot and shares the same tracker/agent seams rather than acting as a separate control plane

## Expected deliverables

- host-mode CLI entry point
- in-memory running/retrying/retired scheduling state
- host-owned max concurrency and max attempts enforcement
- deterministic tests covering retry and redispatch behavior
- docs that explain when to use `poll-once` versus `serve`

## Follow-on scope

- `DEMO-0006` can build on the running-state model for reconciliation
- `DEMO-0007` can project the existing host state into logs and status
- `DEMO-0008` can hot-reload future ticks against the same host state model
