# Execution plan

## Issue

`DEMO-0010`

## Goal

Strengthen the harness safety baseline around workspace paths, external processes, hooks, and secrets without overstating the repository’s current trusted-local posture.

## Recommendation basis

This plan follows the shared harness recommendations that workspace containment and explicit trust boundaries are baseline controls, and that external commands/hooks must be treated as trusted configuration with clear failure behavior.

Key principles:
- workspace-root containment is a non-negotiable invariant
- secret values should not leak through logs, request files, or run records accidentally
- hook/process failure behavior must be explicit, testable, and operator-visible
- the harness should document what it does **not** harden yet rather than implying sandboxing it does not provide

## Prerequisites

- `DEMO-0002` should be complete first.
- Reproduce current behavior around hook execution, request-file creation, process output capture, and run-record writing before tightening rules.

## Bounds

- in scope:
  - revalidate workspace containment before hooks and worker launches
  - review secret exposure paths in logs, transcripts, run records, and generated request files
  - define and document timeout/failure/truncation behavior for hooks and external processes
  - add tests for the tightened safety contract
- out of scope:
  - container or VM isolation
  - claims of network sandboxing or full OS hardening
  - replacing trusted hooks with a completely different extension/plugin model

## Detailed steps

1. **Inventory current safety invariants and leak paths**
   - Review where workspace paths are validated today.
   - Review where external process stdout/stderr is stored or surfaced.
   - Review whether request files, run records, or logs can accidentally include secret env values.

2. **Tighten workspace containment checks**
   - Revalidate the workspace path immediately before hook execution.
   - Revalidate the workspace path immediately before worker process launch.
   - Keep the containment rule centralized so later issues do not bypass it accidentally.

3. **Define explicit process and hook output policy**
   - Decide what is captured in full versus what is truncated.
   - Keep operator logs smaller than workspace-local transcripts where appropriate.
   - Make timeout and non-zero-exit behavior explicit and consistent across hooks and worker commands.

4. **Reduce accidental secret exposure**
   - Ensure secret env values are never echoed in validation or error messages unless explicitly intended.
   - Avoid copying secret-bearing configuration into request files or run records.
   - If redaction is needed, make the redaction rules deterministic and documented.

5. **Add focused tests for safety behavior**
   - Cover path-containment enforcement before execution.
   - Cover timeout/failure behavior for hooks and external processes.
   - Cover any truncation/redaction rules introduced.
   - Prefer deterministic fake commands where possible.

6. **Align docs with actual trust posture**
   - Update docs to explain the strengthened invariants.
   - Be explicit about what remains trusted-local and what is not yet hardened.
   - Avoid implying containerized or remote-safe behavior the repo does not actually have.

## Expected deliverables

- stronger pre-execution workspace containment checks
- documented and tested hook/process timeout/failure/truncation behavior
- reduced accidental secret exposure in operator-visible artifacts
- docs reflecting the real, not imagined, safety posture

## Risks

- risk: safety fixes unintentionally break the proving path or diagnostics
  - mitigation: add focused tests and keep operator evidence useful
- risk: truncation/redaction hides too much information for debugging
  - mitigation: separate operator logs from workspace-local artifacts where needed and document the boundary
- risk: docs imply stronger isolation than the implementation provides
  - mitigation: keep the trusted-local boundary explicit in every related doc update
