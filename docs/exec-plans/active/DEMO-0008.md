# Execution plan

## Issue

`DEMO-0008`

## Goal

Reload `WORKFLOW.md` during long-running execution while keeping the last known good configuration active when a reload fails.

## Recommendation basis

This plan follows the shared harness recommendation that repo-owned workflow policy should remain live and versioned with the codebase, and that a long-running harness should not require restart for routine workflow changes.

Key principles:
- `WORKFLOW.md` remains the authoritative workflow contract
- invalid reloads must be operator-visible without destroying the running service state
- reload behavior should be explicit about what is hot-applied versus restart-required
- the last known good configuration is safer than a partial or broken reload

## Prerequisites

- `DEMO-0005` should establish a long-running host mode.
- Coordinate with `DEMO-0007` if logging/status changes affect how reload failures are reported.

## Bounds

- in scope:
  - detect `WORKFLOW.md` changes during host execution
  - reload supported config/prompt fields without restart
  - keep the last known good config when a reload fails
  - document hot-reload versus restart-required settings
  - test reload success and failure paths
- out of scope:
  - dynamic reload of arbitrary repo files beyond the workflow contract
  - silently applying malformed or partially parsed workflow changes

## Detailed steps

1. **Define the reload contract first**
   - Decide how the host detects workflow changes:
     - periodic mtime/hash checks in the poll loop
     - or a file watcher if justified
   - Prefer the smallest reliable mechanism for a local-first harness.

2. **Define what is hot-reloadable**
   - List which settings are safe to update mid-run, such as:
     - prompt body
     - poll interval
     - concurrency settings for future ticks
     - state lists used for future decisions
   - List which settings remain restart-required if needed.

3. **Implement last-known-good behavior**
   - Keep the currently active workflow/config snapshot in host state.
   - On change detection:
     - attempt reload and validation
     - replace active config only on success
     - leave the old config active on failure
   - Ensure failures are clearly logged or surfaced to the operator.

4. **Clarify how active runs are treated**
   - Decide whether active runs keep the config they started with or can observe updated settings on later turns/ticks.
   - Keep the first version conservative and documented.
   - Avoid silently mutating the meaning of an already running attempt unless the behavior is deliberate.

5. **Test both success and failure paths**
   - Verify that a valid workflow edit is detected and applied.
   - Verify that an invalid edit leaves the prior workflow active.
   - Verify that repeated reload failures remain visible but do not crash the host.

6. **Document the operator contract**
   - Explain what changes are hot-reloaded.
   - Explain how reload failures are reported.
   - Explain what requires restart.

## Expected deliverables

- host-side workflow reload mechanism
- last-known-good workflow retention on reload failure
- deterministic tests for valid/invalid reload behavior
- updated docs describing reload semantics

## Risks

- risk: reload mutates active work in surprising ways
  - mitigation: document and implement conservative active-run behavior first
- risk: invalid workflow edits break the service loop
  - mitigation: keep the last known good config active and log errors clearly
- risk: reload scope becomes ambiguous
  - mitigation: publish an explicit hot-reload versus restart-required list

- the documented list of hot-reloadable versus restart-required settings
