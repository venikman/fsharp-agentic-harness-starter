# Execution plan

## Issue

`DEMO-0001`

## Goal

Make external worker execution a first-class, harness-controlled path while preserving the dry-run proving path and keeping all runtime policy in harness-owned files.

## Recommendation basis

This plan follows the shared harness guidance that separates the orchestrator from the worker runtime:

- the harness owns configuration, workspaces, and run records
- the worker runtime executes one assigned run in one assigned workspace
- repo-owned `WORKFLOW.md` is the configuration surface
- local proving paths should remain available while the real worker path is introduced
- no checked-in `.pi/` settings or other secondary config surfaces are introduced

## Prerequisites

- `DEMO-0002` should be complete first so the repo has a green build and deterministic tests.
- Reproduce the current post-`DEMO-0002` behavior before changing code, because the repository may already contain a partial external-command path.

## Bounds

- in scope:
  - verify and complete the generic external command path in `Agent.fs`
  - ensure token-expanded arguments include workspace and request-file context
  - persist worker stdout/stderr under the workspace `.harness/` directory
  - preserve the `dry-run` path as a supported baseline
  - add tests and/or smoke fixtures for both dry-run and external-command behavior
- out of scope:
  - app-server protocol support
  - multi-turn worker session semantics beyond what this issue needs
  - pi-specific repo config
  - tracker writes, retries, or long-running orchestration logic

## Detailed steps

1. **Re-establish the real gap after the build baseline is fixed**
   - Inspect `src/DeliveryHarness.Core/Agent.fs` and confirm what already works for non-`dry-run` commands.
   - Run the smallest safe smoke check with a benign external command to determine whether this issue is now mostly about validation/docs versus missing behavior.
   - Capture any mismatch between the issue acceptance criteria and the current implementation.

2. **Define the minimum supported generic worker contract**
   - Keep the contract intentionally small:
     - the harness writes `agent-request.md`
     - the harness invokes `agent.command` with token-expanded args
     - the worker runs with `cwd = workspace`
     - the harness captures stdout/stderr into workspace-local artifacts
   - Document the supported token set and do not add speculative templating beyond what the issue authorizes.

3. **Complete or tighten the execution path**
   - Ensure the request file path is always passed when configured in args.
   - Ensure the transcript is written even on non-zero exits or timeouts.
   - Ensure failure summaries are informative but still harness-owned.
   - Preserve the `dry-run` behavior exactly enough that it remains a stable proving path.

4. **Add deterministic validation for the worker path**
   - Prefer a benign local command fixture for tests, such as a shell command that writes a known marker to stdout/stderr.
   - Avoid depending on a real coding-agent CLI in automated tests.
   - Cover:
     - dry-run output path
     - token replacement for `workspace`, `issue_id`, `issue_title`, and `request_path`
     - transcript creation on success and failure

5. **Update docs and workflow expectations**
   - Clarify in docs that runtime configuration remains harness-owned.
   - If the default `WORKFLOW.md` remains `dry-run`, document how to switch to a real worker command explicitly.
   - If a real worker command becomes the default, record the trust and tooling assumption clearly.

6. **Run issue-level validation**
   - Build the CLI project.
   - Run the issue through the harness using either the documented real-worker configuration or the narrowest harness-owned validation setup that proves the external path.
   - Capture workspace `.harness/*` artifacts and the resulting run record.

## Expected deliverables

- completed generic external worker path with preserved dry-run fallback
- deterministic tests or smoke fixtures for worker execution artifacts
- updated documentation of the worker contract and configuration boundary
- review evidence showing request-file passing and transcript persistence

## Risks

- risk: the issue is already mostly implemented and changes accidentally widen scope
  - mitigation: re-establish the exact remaining gap before editing code
- risk: tests depend on a real agent executable
  - mitigation: use benign local command fixtures in automated tests
- risk: worker config drifts into non-harness surfaces
  - mitigation: keep all configuration in `WORKFLOW.md`, docs, and harness-controlled env vars only
