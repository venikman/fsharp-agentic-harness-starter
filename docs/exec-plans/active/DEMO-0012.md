# Execution plan

## Issue

`DEMO-0012`

## Goal

Replace loose prompt concatenation with a small, strict prompt templating contract that uses real issue and attempt metadata and fails fast on unknown variables.

## Recommendation basis

This plan follows the shared harness guidance that repo-owned workflow policy should be explicit, versioned, and trustworthy. A workflow template should describe actual runtime inputs, not rely on hidden string assembly or undocumented conventions.

Key principles:
- `WORKFLOW.md` remains the repo-owned prompt contract
- templates should be small and predictable, not a general programming language
- prompt variables should reflect actual runtime state defined by the harness
- failures on unknown variables are safer than silently producing misleading prompts

## Prerequisites

- `DEMO-0011` should define the real attempt/turn model first.
- Reproduce the current prompt-builder behavior so the migration path is explicit.

## Bounds

- in scope:
  - define a minimal prompt variable surface using issue and attempt context
  - implement strict rendering and fast failure on unknown variables
  - keep a documented compatibility or migration path for the current simple workflow body
  - add tests for rendering and request-file generation
- out of scope:
  - a full templating language
  - arbitrary function calls or scripting inside workflow templates
  - non-harness config sources for prompt behavior

## Detailed steps

1. **Define the minimal variable surface**
   - Base it on real harness state only, such as:
     - issue identity/title/body
     - acceptance/validation/constraints where appropriate
     - attempt metadata defined in `DEMO-0011`
   - Keep the first version intentionally small.

2. **Specify the strict rendering contract**
   - Decide on the placeholder syntax.
   - Define what counts as an unknown variable.
   - Make validation and/or run-start fail clearly when templates reference unsupported variables.

3. **Design the compatibility path**
   - Decide whether existing workflow bodies without template variables:
     - continue to work unchanged
     - or are migrated through a documented fallback mode
   - Keep migration behavior explicit; do not silently combine incompatible modes.

4. **Implement rendering at the right seam**
   - Keep prompt rendering harness-owned and close to request-file construction.
   - Avoid leaking templating concerns into unrelated orchestration code.
   - Ensure the generated request file reflects the fully rendered prompt.

5. **Add deterministic tests**
   - Cover successful rendering with known variables.
   - Cover unknown-variable failure.
   - Cover compatibility/fallback behavior if retained.
   - Cover request-file output shape so later worker/runtime work can rely on it.

6. **Document the template contract**
   - Update workflow or docs with the supported variable list.
   - Make clear that the template surface is intentionally limited.
   - Keep the explanation harness-owned and repo-versioned.

## Expected deliverables

- strict prompt rendering contract
- minimal supported variable set based on real harness state
- tests for success, failure, and compatibility behavior
- docs describing how to author `WORKFLOW.md` templates safely

## Risks

- risk: templating grows into a mini-language that is hard to reason about
  - mitigation: keep the first variable surface tiny and declarative
- risk: compatibility behavior becomes ambiguous
  - mitigation: document one explicit compatibility/migration path and test it
- risk: templates reference runtime fields that do not actually exist
  - mitigation: base the variable list only on already-defined harness state from prior issues
