# Execution plan

## Issue

`DEMO-0004`

## Goal

Introduce a tracker abstraction that keeps the markdown file tracker as the default proving adapter while creating the seam required for future tracker integrations.

## Recommendation basis

This plan follows the shared recommendation that the orchestrator should depend on a tracker integration boundary rather than tracker-specific details.

Key principles:
- the harness owns issue orchestration, not tracker vendor details
- the file-backed tracker remains the trusted local proving path
- the tracker seam should cover only the responsibilities the orchestrator actually needs
- tracker writes remain out of scope for the orchestrator core in this pass

## Prerequisites

- `DEMO-0002` should be complete first.
- Reproduce the current `FileTracker` call graph from CLI and orchestrator modules before refactoring.

## Bounds

- in scope:
  - define the minimal tracker port/interface required by current orchestration
  - adapt the markdown tracker to that boundary
  - update CLI and orchestration code to depend on the seam instead of `FileTracker` directly
  - preserve existing issue markdown format and behavior
- out of scope:
  - networked tracker integration itself
  - tracker writes such as comments or state transitions
  - unnecessary abstraction for features the harness does not currently use

## Detailed steps

1. **Map the current tracker responsibilities**
   - Identify the concrete responsibilities already in use:
     - list candidate issues
     - find issue by id
     - refresh state for reconciliation-style behavior
     - fetch terminal issues for cleanup logic
   - Use the current code paths in CLI and orchestrator as the source of truth.

2. **Define the minimal tracker boundary**
   - Create a small abstraction that matches current and near-term needs only.
   - Keep the normalized issue model harness-owned, not adapter-owned.
   - Prefer explicit functions over a prematurely large service surface.

3. **Adapt the file-backed implementation**
   - Wrap or refactor `FileTracker` so it implements the new seam.
   - Preserve current sorting, parsing, and issue-loading behavior unless an issue-approved fix is required.
   - Ensure `list-issues` and `run-issue` still work without workflow changes.

4. **Refactor orchestrator and CLI call sites**
   - Route current tracker access through the abstraction.
   - Keep the workflow-driven tracker selection harness-owned.
   - Avoid coupling the abstraction to long-running orchestration concerns that belong to later issues.

5. **Add tests for the boundary, not just the implementation**
   - Verify the file-backed adapter still supports:
     - candidate enumeration
     - lookup by id
     - sorted issue ordering
     - state-based filtering inputs needed by orchestration
   - Add targeted tests for the new seam so later adapters have a behavior contract.

6. **Document the new seam**
   - Update architecture docs if the module ownership or dependency direction changes.
   - Clarify that the file-backed adapter remains the default proving path until later tracker issues land.

7. **Run issue-level validation**
   - Build the CLI project.
   - Run `list-issues` and confirm output still reflects the markdown tracker.

## Expected deliverables

- a minimal tracker abstraction in core code
- file-backed adapter preserved behind the new seam
- updated orchestrator/CLI usage of the seam
- tests that describe tracker behavior expected by the harness
- docs clarifying that later adapters plug into this boundary

## Risks

- risk: abstraction becomes larger than the harness currently needs
  - mitigation: model only current orchestration responsibilities plus near-term reconciliation/cleanup needs
- risk: file-backed behavior changes accidentally during refactor
  - mitigation: add regression tests around issue parsing/order and CLI-visible behavior
- risk: later tracker integration pressure leaks vendor concerns into the core model
  - mitigation: keep normalized issue types and orchestration decisions harness-owned
