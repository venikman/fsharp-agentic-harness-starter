# Execution plan

## Issue

`DEMO-0012`

## Status

Completed baseline reference retained until plan archival exists.

## Outcome

Workflow bodies now support strict `{{ ... }}` prompt variables for documented issue/attempt/turn fields, unknown or malformed markers fail validation, and workflows without markers keep the legacy wrapped issue-context compatibility path.

## What later work should assume

- the prompt-template contract is small, explicit, and already real
- strict templates support only documented `issue.*`, `attempt.number`, `turn.number`, and `tracker.kind` variables
- unknown variables and malformed markers fail validation and run start
- compatibility mode remains available for simple workflow bodies without template markers

## Expected deliverables

- strict prompt rendering contract
- documented supported variable set
- compatibility mode for legacy workflow bodies
- deterministic validation for success, failure, and request-file rendering

## Follow-on scope

- later prompt work should extend the variable list conservatively rather than introducing a mini-language
- future worker/runtime work may add new variables only when the underlying runtime state is already real
