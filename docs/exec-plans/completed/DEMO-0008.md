## Outcome

Host mode now checks `WORKFLOW.md` mtime on each tick, hot-reloads the documented future-tick settings, and keeps the last known good workflow active when reloads are invalid or restart-required.

## What later work should assume

- workflow reload is already a real host-mode behavior
- active runs keep the workflow they were dispatched with
- prompt body, state sets, cleanup policy, and orchestrator cadence/concurrency/retry settings are hot-reloaded for future ticks/runs
- tracker location, workspace root, worker command/args/timeout, `agent.max_turns`, and hooks remain restart-required

## Expected deliverables

- per-tick workflow change detection
- last-known-good workflow retention on invalid reload
- deterministic tests for valid and invalid reload behavior
- repo docs that publish hot-reloadable versus restart-required settings

## Follow-on scope

- later continuation or tracker work should preserve the same last-known-good rule
- any richer reload behavior should stay conservative for already-running attempts
